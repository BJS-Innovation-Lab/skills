# Learning Insights — March 7, 2026

## Railway webhook integration strategy
**Context:** Building agent messaging system for distributed VULKN fleet

**Learning:** Railway agents don't need separate notifier processes. OpenClaw has a built-in `/hooks/agent` endpoint that receives webhooks directly on the main port (8080). This is cleaner than TCP proxies which can conflict with the main application.

**Stakes:** medium (wrong approach wastes time and creates deployment complexity)

## VibeKanban architectural limitation
**Context:** Johan researching VibeKanban for coding agent orchestration

**Learning:** VibeKanban is designed for LOCAL coding agents only (Claude Code, Cursor on developer's machine). Cannot connect to remote agents on Railway or other distributed infrastructure. For distributed agent fleets, need to build custom orchestration.

**Stakes:** medium (prevents wasted integration effort on incompatible tool)

## Memory system architecture insight
**Context:** Could not find WhatsApp Business credentials despite setting them up weeks ago

**Learning:** Daily memory logs are episodic (what happened) but missing semantic indexes (what I know). Need separate files for service/infrastructure state that persist across sleep-consolidation cycles. Script-based memory generation produces noise - agent-driven memory refresh understands context and writes useful summaries.

**Stakes:** medium (causes "credential amnesia" and repeated setup work)

## Agent correction patterns as training data
**Context:** Building reward hacking detection corpus with Bridget for Harvard ML project

**Learning:** Agent conversation logs where humans say "no you didn't" or correct reported success are natural labels for reward hacking detection. Don't need to manufacture synthetic examples - real correction patterns from agent interactions provide high-quality training data.

**Stakes:** low (useful for ML research but not critical for operations)

## Cloud agent naming convention
**Context:** Setting up Sam Cloud agent - conflicts with local Sam agent

**Learning:** Cloud agents need distinct AGENT_IDs from their local counterparts to avoid messaging conflicts. Use pattern like `sam-cloud` vs `sam` or `sybil-cloud` vs `sybil` for clear separation.

**Stakes:** medium (prevents message routing conflicts between local and cloud versions of same agent)
