# Learning Corrections — March 7, 2026

## Railway webhook body-parser bug
**Context:** Setting up webhooks for Vulki agent on Railway - POST requests to `/hooks/agent` were hanging with timeouts

**Learning:** The `express.json()` body-parser middleware at line 300 in the Railway wrapper was consuming the request body before the proxy could forward it to OpenClaw. The middleware exempted `/v1/` paths but NOT `/hooks/` paths. This caused the gateway to hang waiting for a body that was already consumed.

**Fix:** Add `/hooks` exemption to body-parser: `if (req.path.startsWith("/v1") || req.path.startsWith("/hooks")) return next();`

**Stakes:** high (blocks all webhook functionality for Railway agents, causing silent failures)

## Agent messaging addressing bug
**Context:** Setting up agent messaging system - some agents weren't receiving messages sent via A2A

**Learning:** The `agent_messages` table had inconsistent addressing - some messages used agent names (`saber`, `sybil`) while others used UUIDs (`415a84a4-af9e-4c98-9d48-040834436e44`). Inbox scripts typically filter by name, so UUID-addressed messages were getting missed.

**Fix:** Always use agent **names** (not UUIDs) in `from_agent` and `to_agent` fields.

**Stakes:** medium (causes missed communications between agents)

## Critical Supabase URL typo in documentation
**Context:** Setting up Scout cloud agent - connection to HQ Supabase was failing

**Learning:** Cloud agent setup documentation had wrong Supabase URL suffix. Used `.com` instead of `.co` which caused all database connections to fail silently.

**Fix:** WRONG: `https://fcgiuzmmvcnovaciykbx.supabase.com` → RIGHT: `https://fcgiuzmmvcnovaciykbx.supabase.co`

**Stakes:** high (completely breaks all cloud agent database connectivity)
