# Learning Corrections: 2026-03-04

## Learning Extraction Scripts vs Agent Judgment
**Context:** Fixed learning extraction system that was missing 97% of learnings
**Learning:** The old `extract-insights.cjs` used strict regex that caught only 2 learnings vs 10 found by agent judgment. Agent-based reading of daily notes is far more effective than rigid pattern matching.
**Stakes:** high (without proper learning extraction, I repeat mistakes and forget important insights)

## Git Safety for Destructive Operations
**Context:** Enforced git safety rules after Feb 19 incident where rebasing lost 453 files  
**Learning:** Always run `git diff --stat HEAD~N` before any `git reset`, and `git stash` uncommitted changes. Never use `git clean -fd` without understanding what will be deleted.
**Stakes:** high (data loss incidents break productivity and lose work)

## HQ vs Field Agent Data Isolation
**Context:** Built memory search architecture that initially gave field agents access to all HQ agent documents
**Learning:** Data isolation requires careful Supabase query filtering. HQ agents can see all documents (no agent_id filter), field agents should be local-only for client data privacy.
**Stakes:** high (client data leakage is a major privacy/security issue)

## A2A Relay URL Coordination  
**Context:** Multiple agents (Sage, Vulki) had 404 errors from outdated A2A relay URLs
**Learning:** A2A infrastructure changes require coordinated updates across all agents. The canonical URL is now `a2a-bjs-internal-skill-production-f15e.up.railway.app` (note the `-f15e` suffix).
**Stakes:** medium (broken A2A prevents team coordination and task routing)

## Etsy API Manual Approval Process
**Context:** Set up Etsy developer app for StorybookRetro but API key was pending
**Learning:** Etsy requires manual approval for new API applications - it's not automatic. Need to plan for approval delays in client work.
**Stakes:** medium (blocks client work and automated scraping fallbacks)
